## ① 阅读生词本按 A–E 频率档着色 + 去掉左侧竖色条

**现状**（`index.html`）：右侧阅读生词本（`renderNbList`，~L2027）用 `nbLvl(r)`（~L2014）算一个颜色，**跟着排序维度变**（clicks→点击色、order→章节色、freq→频率色），并通过 `data-lvl` + `style="--lvl:${col}"` 渲染成**左侧竖色条**：
- `.nb-item{border-left:3px solid transparent;padding-left:8px}` (L394)
- `.nb-item[data-lvl]{border-left-color:var(--lvl)}` (L395)
- 词文字 `.nb-item .nw{...color:var(--fg)}` (L377) —— 现在恒为深色。

对照：**全局**生词本 `gvFreqVar(r)`（L2176）＝「整行文字永远染频率色（A→E），与排序无关」，无左色条。用户要阅读生词本对齐它。

**改法**：
1. **词文字染频率色**：把 `.nb-item .nw` 的 `color:var(--fg)` 改为 `color:var(--lvl,var(--fg))`。词块/句子/段落无频率档 → `--lvl` 不设 → 回落 `--fg`（可接受；如想更贴齐全局的「无档=muted」，phrase/sentence/para 行给 `--lvl:var(--muted)`，由你在 `nbLvl` 决定，见下）。
2. **着色恒为频率、与排序解耦**：把 `nbLvl(r)` 改成**永远按频率档**取色（词：`{A:'--fA'...E:'--fE'}[r.freq_band]`；非词或无档：返回 `''` 或 `--muted`）。删掉它里面 `_nbSort==='clicks'`/`==='order'` 的分支——排序仍能改顺序，但**颜色不再跟排序变**（对齐全局「拍板」行为）。
3. **去掉左侧竖色条**：删 `.nb-item{border-left:3px solid transparent;padding-left:8px}` 的 border-left（padding-left 可保留或按视觉微调）＋删 `.nb-item[data-lvl]{border-left-color:var(--lvl)}`。`data-lvl`/`--lvl` 变量仍要保留在 `.nb-item` 上（供 `.nw` 文字取色）。
4. **文案**：`sortSeg` 的 title「排序维度（左色条随之分层着色）」(L2038) 改成「排序维度」（色条已去除，别留误导文案）。

**验收**：preview harness 打开生词本，`getComputedStyle('.nb-item .nw')` 三种排序下颜色都＝对应词的频率色且**切换排序颜色不变**；`.nb-item` 无 `border-left` 可见色条；截图与全局生词本并排观感一致。

## ② 整段高亮自动进生词本 +「段落」筛选档（段落只存原文）

**现状**：`applyHighlight(range,colorKey)`（~L2697）检测选区 trim 后是否等于所在 block 全文，命中走 `_applyParaCard`（`.hl.para` 整段色卡，L2677），record 入 `_highlights`（`kind:'para'`）。**但不写生词本**。生词本 kind 只有 word/phrase/sentence（`NB_TAG` L2012、筛选 chips L2040）。

**改法**：
1. **后端 `app.py` 新增 `collect_paragraph(args)`**（照抄 `collect_keyword` 的形态，~L716）：入参 `{text}`（整段原文）。**不调 LLM**，直接写一条 notebook 词条：`kind="para"`，`§`-key 幂等去重（可用 `"¶"+归一化文本` 或沿用 `§` 前缀+hash，键形态自定但要与 word/phrase 键不撞、且同段重复收藏去重），字段 `text=原文`、`meaning=""`。并 `_upsert_global` 汇入全局（与 keyword 一致）。回 `{ok, key, collected:bool}`。
   - `save_session`/`load_archive`：`para` 条目本就在 `_notebook` 里，随 notebook 一并存取，无需额外字段。
2. **前端 `applyHighlight` 整段命中分支**：在 `_applyParaCard(...)` 成功后，`await api().collect_paragraph({text: 整段原文})`，成功则 `scheduleSave()` + `refreshNbCount()`。整段原文＝该 block 的 `textContent`（trim）。
   - 注意：`applyHighlight` 有两处 push `kind:'para'`（L2713、L2768，一处新建一处复用上次色），**两处都要触发收藏**——抽一个 `_collectPara(text)` helper 调用，避免漏。
3. **生词本渲染支持 para**：
   - `NB_TAG` 加 `para:'段'`（L2012）；`nbKindOf` 让 `para` 归为自身一类（L2013 改成识别 phrase/sentence/para）。
   - 筛选 chips（L2040）加 `${chip('para','段落')}`；`counts` 初始化加 `para:0`（L2034）。
   - `para` 行标题：整段原文过长，沿用现成的 `head=disp.length>72?...` 截断（L2059）已能处理；展开时 `mountExplanation` 显示全文 + 可追问。
   - para 无频率档 → 不着色（`--muted`），与词块/句子一致。
4. **点 para 条目可回原文定位**（可选但推荐，若成本低）：`removeArticleMark`/`locateInArticle` 目前按 lemma/§。para 用文本匹配定位到对应 `.hl.para` block（best-effort，找不到就算了，不报错）。

**验收**：preview harness 划选整段→上整段色卡→生词本计数 +1、出现「段」标签条目、原文完整存入；重复收藏同段不增计数（幂等）；`get_notebook` 形态含 `kind:'para'`；「段落」筛选档只显示 para 条目。Python 侧 `collect_paragraph` 幂等/空拒/全局汇入断言。

## ③ 书页上「讲解带 / 收藏带」(`.phrase`) 跨换章稳定必现

**现状**：`.hl` 色卡高亮**已有**跨章锚点持久（`_highlights` 数组 L3786、`_applyHighlightsForChapter` L3830、`save_session`/`load_archive` 的 `highlights` 字段 app.py L1426/L3941）。但 `.phrase`（划选讲解 `explainSelection`→`wrapPhrase` L2589 生成的语块带；以及收藏 keyword 时在原文标出的语块）在**书模式没有对应锚点数组**——`save_session`（L3173）只存 `bookmarks/highlights/dots`，**没存 phrases**。故换章回来 `.phrase` 带消失（文章模式靠 innerHTML 快照天然持久，不受影响）。

**改法（照抄 `_highlights` 那一整套，给 `.phrase` 做平行实现）**：
1. **前端锚点数组 `_phrases`**（镜像后端）：每个 record `{ch, blockOrd, start, end, text, sentence, kind:'phrase', ...携带 data-ex 里追问/讲解所需的最小字段}`。start/end 用现成的块内字符偏移工具（`_hlPointToBlockOffset`/`_hlOffsetToPoint`，TreeWalker 累加，L3784 附近），与 `_highlights` 同一套几何。
2. `wrapPhrase` 在书模式额外算锚点 push 进 `_phrases`（参考 `applyHighlight` 书模式分支 L2737 push `_highlights` 的写法）；文章模式行为不变（不 push，仍走快照）。
3. **新增 `_applyPhrasesForChapter()`**（照抄 `_applyHighlightsForChapter` L3830）：`renderChapterContent` 渲完、且在 `_applyHighlightsForChapter()` 之后调用，按 `_phrases.filter(p=>p.ch===bookIdx)` 用 Range 重新包 `.phrase`（复用 `wrapPhrase` 的 surroundContents 技术；把 data-ex 还原到 `dataset.ex` 使追问历史随之复原）。
4. **后端 `app.py`**：`save_session`/`load_archive` 书模式分支各加 `phrases` 字段，逐字照抄 `bookmarks`/`highlights` 的 plumbing（+3 行量级）。旧档无该字段按空数组兼容。
5. 前端 `save_session` 调用处（L3173）参数加 `phrases:_phrases`；`load_archive`/`renderBook` 载入处（L3941 附近）`_phrases=Array.isArray(r.phrases)?r.phrases:[]`。

**注意**：调用点选在 `renderChapterContent` 内容赋值之后（与 `_highlights` 同因——避开 resize 触发 `_initPager` 时的重复包裹）。`.phrase` 内若嵌 vocab 词仍要可点（与 `.hl` 嵌套同理，验证一下）。

**验收**：preview harness 书模式划选讲解生成 `.phrase`→record 入 `_phrases`；`gotoChapter` 换到别章再回来，`.phrase` 从 DOM 消失后**恢复**（数量、文本、data-ex 追问历史都在）；`save_session→load_archive→renderBook` 全链路后 `_phrases` 正确带出、重渲后 `.phrase` 仍在。文章模式零回归（仍走快照）。

## ④ 追问后台化：自由切 a/b/c… 谁的追问都不丢、都在后台续跑、回看必复原

**现状根因（已定位，别再猜）**：
- 点新词时 `explain()`（L1448）`panelEx.innerHTML=loading` → `showExplanation` 整块替换 panelEx。此时上一个词在途的 `.ask` 块 + 捕获的 `aEl` 被**从 DOM 摘除**。
- `sendAsk`（L1625）的请求 **`llmCall` 不做 abort**（L1801，只计 busy 数）——后台照跑到底，后端 `ask_followup`（app.py L773）**已经把答案写进 notebook 条目**（L808 `entry.setdefault("followups",[]).append(...)`）。所以答案没真丢，只是①落进被摘除的旧 DOM 看不见、②回看时主面板不复原。
- 回看不复原的原因：`explain_word`（app.py L603）返回的 `result` **不带 followups**（followups 只写进 `entry`/notebook，没并进 `result`；`_cache` 也从不含 followups）。故再点该词，`mountExplanation` 读 `r.followups||[]`＝空。`explain_selection`（L668）同理。

**改法**：
### 4a. 后端让讲解结果带上 followups（回看必复原的地基）
- `app.py explain_word`：`return result` 前，把该词 notebook 条目里的 `followups` 并进 `result`（`entry.get("followups")` 存在则 `result["followups"]=entry["followups"]`）。
- `app.py explain_selection`：同样把对应 `§`-key 条目的 `followups` 并进返回值。
- 这样前端每次点词/短语，`mountExplanation`（L1602）都能用 `r.followups` 还原已完成的追问线程。

### 4b. 前端「在途追问登记表」+ 切回自动复原/续接
- 新增模块级 `const _pendingAsks = new Map();` key＝该词/短语的稳定标识（`ctx.lemma || ctx.word` 归一化小写；短语用 `§`-key，与后端一致）。value＝该 key 下**在途**追问数组：`[{label, promise, status:'pending'}]`。
- `sendAsk`（L1625）发起时：把 `{label, promise, status:'pending'}` 登记进 `_pendingAsks[key]`；`promise` 完成（成功/失败）后从数组移除，并把结果并进该 key 的客户端 followups 缓存（供无需再请求后端就能复原；后端也已持久）。**即便 `block`/`aEl` 已被摘除，promise 的 .then 也不能抛错**（判断 `aEl.isConnected` 再写 DOM）。
- `mountExplanation`（L1602）渲染某词时：除了用 `r.followups` 渲染已完成历史，还要检查 `_pendingAsks[key]`——对每条 pending 追问，在 thread 里补一个「问句气泡 + 活的 spinner」，并把它的 `promise.then` **重新绑定到新建的 aEl**（当且仅当此刻面板仍显示该 key 时写入；用一个「当前面板 key」标记 `_panelKey` 判断）。这样：切走→在途继续；切回→看到还在转的 spinner，完成时自动填答；已完成的从 followups 复原。
- **验证请求确实不被取消**：切词后原 promise 仍 resolve、后端 notebook 仍收到该条 followup。

**判断题（你需要处理的边界）**：同一词并发多条追问要都能各自归位（用数组，不是单值）；切回时不要把同一条历史渲染两次（followups 已完成的 与 _pendingAsks 在途的 不重叠——完成即移出 pending）。

**验收（preview 闭环，这是最容易错的一项，量数字）**：
1. 对词 A 发起追问（mock `ask_followup` 延迟 resolve）→ 未 resolve 时点词 B → A 的 promise 仍在、B 正常显示讲解；
2. A 的 promise resolve 后，后端/客户端缓存 followups +1（数字）；
3. 切回 A：thread 里能看到 A 的那条追问答案（已完成复原）或仍在转的 spinner（若未完成）；
4. 连续 A→B→C→A 来回切，无报错、无重复渲染、无答案丢失；
5. `console_logs` 全程零报错。

## ⑤ 追问连续性放宽（学明白优先）

**现状**：连续性**已有**（`llm.py followup` L256 拼多轮 messages：system + 语境含 prior + history 每轮 user/assistant + 新问）。两个前端/后端设的窗口偏紧：
- 前端 `sendAsk`（index.html L1636）`history:ctx.history.slice(-6)` —— 只发最近 6 轮。
- 后端 `llm.py` L280 `prior[:500]` —— 讲解截 500 字。

**改法**（用户明确「目的是学明白，不是省钱」）：
- L1636：`slice(-6)` → `slice(-40)`（等于「放开但仍有防跑飞兜底」，40 轮实际几乎不可能 hit）。
- `llm.py` L280：`prior[:500]` → `prior[:2000]`。
- 就这两个数，别的不动。

**验收**：`node --check` + `py_compile` 过；构造 8 轮历史，确认第 8 问的 messages 里带齐前 7 轮（不再被 6 砍掉）。
