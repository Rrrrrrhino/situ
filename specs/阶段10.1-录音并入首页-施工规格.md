## 1. Swift「四土对话录」加 headless 模式

- `main.swift`：`CommandLine.arguments` 含 `--headless` 时走无 UI 路线，否则走现有菜单栏路线（零改动）。
- **headless 路线**（新文件 `HeadlessRunner.swift`，仍经 NSApplication + delegate 的
  didFinishLaunching 启动——别在 run() 前建任何东西，阶段10 的启动竞态教训）：
  1. `NSApp.setActivationPolicy(.prohibited)`（纯后台，无 Dock 无菜单栏）。
  2. didFinishLaunching 里直接 `recorder.start()`。
  3. RecorderDelegate：
     - `recorderDidStart` → 把 `{"pid": <pid>, "dir": "<会话目录名>", "startedAt": "<ISO8601>"}` 写到
       `~/Documents/situ/data/dualtrack/.recorder.pid`（JSON）。
     - `recorderDidStop(dir)` →（Recorder 已落 ready）删 .recorder.pid → `exit(0)`。
     - `recorderDidFail(error)` → 把「时间 + error.localizedDescription」写
       `data/dualtrack/.recorder.error`（覆盖写）→ 删 .recorder.pid → `exit(1)`。**不弹 NSAlert**
       （headless 下 alert 无人看见还挂进程；菜单栏模式的 alert 保留不动）。
  4. SIGTERM/SIGINT：`DispatchSourceSignal`（先 `signal(SIGTERM, SIG_IGN)`）→ `recorder.stop()`
     → 等 recorderDidStop 正常收尾退出。2 小时安全上限触发的自动停同样走 recorderDidStop → exit(0)。
- 菜单栏模式行为一个字不变。

## 2. server.py：三个录音控制端点

- **App 路径解析**（抽一个 `_recorder_app_path()`，将来 dmg 打包只改①）：
  ① 环境变量 `SITU_RECORDER_APP`（存在且有效则用）；② `<项目根>/对话录/dist/四土对话录.app`；
  ③ `~/Desktop/四土对话录.app`。都不存在 → 端点返回真实错误。
- `POST /api/recorder_start`：
  - .recorder.pid 存在且 pid 活着（校验 `ps -p <pid> -o command=` 含 `DualTrackRecorder` 防 pid 重用）
    → `{ok:false, error:"已在录音中"}`。pid 文件存在但进程死 → 视为脏文件，清掉继续。
  - 清掉旧 .recorder.error → `open -n "<app>" --args --headless`（用 open 而非直接 exec：
    让 TCC 权限记在「四土对话录」名下，与谁拉起它无关）→ 轮询等 .recorder.pid 出现（≤5s，间隔 0.2s）
    → `{ok:true}`；超时 → 读 .recorder.error（有则带真实原因）→ `{ok:false, error:...}`。
- `GET /api/recorder_status`：
  - pid 文件存在且进程活 → `{recording:true, startedAt, elapsedSec}`（elapsed 由 startedAt 现算）；
  - 否则 `{recording:false}`，若 .recorder.error 存在附 `error` 字段（读后不删，start 时才清）。
- `POST /api/recorder_stop`：
  - 读 pid（同样校验 argv）→ SIGTERM → 轮询等进程退出且该会话目录 ready 出现（≤10s）→
    `{ok:true, dir}`；超时 → `{ok:false, error:"停止超时，录音可能仍在进行"}`（不 SIGKILL，宁慢不丢数据）。
- 三端点都只碰 `data/dualtrack/` 下内容；错误信息如实透传。

## 3. 首页 UI（app.js renderHome + style.css + localapi.js）

- `localapi.js` 加三个薄封装：`recorder_start()/recorder_stop()/recorder_status()`（fetch 对应端点，
  网络失败抛出而非吞掉）。
- `renderHome()` 顶部、`'书架'` sec-h 之前插「对话录音」卡：
  - 渲染前先 `recorder_status()`；**请求失败（手机版无 server）→ 整卡不渲染**，书架照旧。
  - **未录音态**：细线麦克风 SVG（照底部 nav 的 1.6 stroke 风格，别用 emoji）+ 标题「对话录音」+
    小字「和 AI 语音对话时录下双轨，聊完来复盘」+ 深蓝主按钮「开始录音」。
  - **录音中态**：红点呼吸（CSS animation，复用项目已有节奏）+「已录 M:SS」+ 按钮变「停止录音」。
    每 1s 轮询 status 刷新计时；**态由 status 驱动**——页面刷新/重进 app 也能恢复录音中态。
  - 停止成功 → toast「录音已保存」+ 卡上出现「去复盘 →」链接（`data-nav` 跳 review tab）。
  - 失败 → toast 真实 error；error 文案含「屏幕录制」「麦克风」「权限」任一关键词时，追加一行
    「去 系统设置 → 隐私与安全性 开启，然后重试」。
  - status 返回 error 字段（上次失败残留）→ 卡上小字灰显上次失败原因，不挡开始按钮。
  - 样式复用项目暖纸卡与既有色板变量（深蓝主按钮、红点用已有警示红），**不新造颜色**。
- 轮询只在 home 屏活跃时跑（切走 tab 清 interval，防泄漏）。
