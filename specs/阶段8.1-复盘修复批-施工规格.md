## 1. 修录音转写的诚实性 + 防数据丢失（review.js）

**已定位的缺陷**（主会话取证）：`_stopRecord`（~1255 行）在 `await _flushSegment(st, true)` 后
**无条件** `toast('已转写')`；而 `_flushSegment` 有三个失败/空文本 return 路径（~1220/1225/1229），
全被这句假成功盖掉。且 flush 开头就清空 `segInt16Chunks`，转写失败=几分钟录音直接丢，无法重试。

改法：
1. `_flushSegment` 返回结果对象 `{ok:boolean, appended:boolean, error?:string}`（成功且有文字追加
   =ok+appended；火山没听清空文本=ok+!appended；请求失败=!ok+error）。
2. `_stopRecord` 按真实结果提示：appended→「已转写」；ok 但空→「没听清，请靠近麦克风重录」；
   失败→「转写失败：{error}」。
3. **失败不丢音频**：`_flushSegment` 失败时把该段 `wavB64` 存入模块级 `RS._pendingWav`
   （只保留最近一段），录音按钮变为「重试转写」态（细线图标风格照旧）；点击→直接用
   `RS._pendingWav` 重新调 `transcribe_audio`（不重录），成功后追加 textarea、清 pending、按钮复位；
   再失败仍保留可再试。用户开始新录音时清掉 pending。
4. 凭证缺失的失败文案要指路：localapi 桌面分支查 IndexedDB 无 `volc_appid`/`volc_token` 时，
   错误信息明确为「未配置火山转写凭证：请到设置 → 语音转写 里填 AppID 和 Token」（若现有文案已
   如此则不动）。

## 2. deepseek-chat 弱档别名升级映射（llm.js）

背景：`deepseek-chat` 别名已被官方悄悄映射到弱档 v4-flash（全局 memory 坐实）。用户各端 IndexedDB
里存的旧 model 值可能是 `deepseek-chat`，逐端手改不现实，在代码层统一升级：

- `_loadConfig` 与 `_loadReviewConfig` 里：当 `provider==='deepseek'` 且解析出的 model 为
  `deepseek-chat` 时，**返回值改用 `deepseek-v4-pro`**（不改写存储值，只在加载时映射；console.log
  一行说明发生了映射，便于取证）。
- 设置屏 model 输入框的 placeholder/说明若提到 deepseek-chat，改为 deepseek-v4-pro 并注明
  「deepseek-chat 已是弱档别名，会被自动升级为 v4-pro」。
- 其它 provider 不动。
