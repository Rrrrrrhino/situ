> 给 builder(Sonnet) 冷启动看的施工单。只读本文件即可施工。
> 分两部分：第一部分（①–④）小而独立，先做完先验收；第二部分（⑤目录切章）较大，紧随其后。

## 1. 背景 / 目标
第一批书签真机验收通过后用户提出的体验微调与三项既有问题：放/删书签的状态小字多余；点空白确认窗口焦点会误放书签；讲解预生成缓存不落盘导致同一页每次重启都重新调 LLM（费钱）；启动长时间白屏；《To Kill a Mockingbird》类 epub（少量巨型 HTML 打包整本书）目录错乱。

## 4. 第一部分（①–④）

### ① 去掉放/删书签的状态小字
- 放置成功、删除成功都**不再 setStatus**（书模式+文章模式共 4 处左右）。其它 toast（如朗读错误）不动。
- 验收：harness 放/删书签 → `statusEl` 文本始终为空。

### ② 防误触（用户痛点：点空白只是想让四土成为当前窗口，却放了书签）
两道闸，同时上：
- **焦点保护**：`window.addEventListener('focus', ...)` 记录 `_winFocusTs=performance.now()`；书签的**放置与点线删除**两条路径开头判 `performance.now()-_winFocusTs < 450` 则直接 return（这次点击只当作激活窗口）。注意 app.py 上一批打了 acceptsFirstMouse 补丁，激活那一击现在会进网页，这道闸就是接住它的。页面首次加载也初始化 `_winFocusTs`，防冷启动第一击误放。
- **间隙限定**：放置点击的 y 必须落在「真实空白」——不在任何文本 fragment 矩形内部（行内空白/字间空白不算数）。即只有段间空隙、列尾空白才放。实现：命中边界后再校验点击点不在任何 frag rect 内（`_bkBoundaries` 已有 frag 数据，代价 O(n)）。
- 验收（harness，量数字）：a) 模拟 `window.dispatchEvent(new Event('focus'))` 后 100ms 内点段间空白 → 记录数不变；450ms 后再点 → 正常放置。b) 点段落文本行内的空白（两词之间）→ 不放；点段间空隙 → 照常放。c) 点线删除同样受焦点保护（focus 后 100ms 点线 → 不删）。

### ③ 启动白屏（根因：head 里同步阻塞的 Google Fonts `<link>`，用户网络走代理，慢/断时首屏一直白）
- 把 Spectral(400/500/600/斜400) 与 Bitter(400/500/600/斜400) 的 woff2 下载到 `santu_app/assets/fonts/`（Google Fonts 可下；若网络不通，用 `curl -x http://127.0.0.1:7897` 走代理再试），改成本地 `@font-face` + `font-display:swap`，**删除** fonts.googleapis/gstatic 的 `<link>`（含 preconnect）。字重子集只要 latin 即可（这两个是西文字体）。
- `webview.create_window(...)` 加 `background_color="#f7f2e8"`（暖纸底）——残余的加载空窗不再刺眼纯白。
- 验收：a) `grep -c "fonts.googleapis\|fonts.gstatic" index.html` → 0；b) preview 打开 harness，`preview_network` 无对 googleapis 的请求，正文 computed font-family 首选 Bitter 且实际生效（`document.fonts.check('17px Bitter')` → true）；c) `create_window` 参数 grep 证据。字体文件总体积汇报出来（预期 <1.5MB）。

### ④ 讲解预生成缓存持久化（用户主诉：同一页每次重启都在「生成讲解」，费钱）
现状：`Api._cache`（lemma→讲解 dict）纯内存。改为**按书落盘**：
- 文件：`DATA_ROOT/pregen/{doc_id}.json`，结构 `{"meta":{"level":..., "ver":1}, "cache":{lemma: 讲解dict}}`。
- 时机：打开书/文章（process / load_archive / load_chapter 共用的初始化处）若存在且 `meta.level == 当前难度` 则整体载入 `_cache`；`_cache` 每新增（讲解成功、pregen worker 写入）后**去抖 2s** 原子写盘（复用 `_save_global` 的原子写手法 + 独立 lock）；`closing → os._exit(0)` 前先同步 flush 一次（在 closing handler 里调，注意此时不能再起线程）。
- level 不匹配 → 忽略旧文件（重新生成后覆盖写）。词块/句子讲解若走同一 `_cache`（key 形态不同）一并受益，无需区分。
- 验收（python 脚本证据）：a) mock explainer 填 3 条 → flush → 新建 Api 实例同 doc_id 同 level → `explain_word` 命中缓存且 **mock LLM 未被调用**（LLM stub 一被调就 raise）；b) level 改变 → 旧缓存被忽略；c) 写盘文件为合法 JSON 且原子写（临时文件+rename）。

## 5. 第二部分：⑤ 目录切章（按 epub 真实 TOC 拆分巨型章节）
**取证结论（主会话已验）**：`books/cb089c878a12.epub`（To Kill a Mockingbird）spine 只有 6 个巨型 HTML（每个 ~1000 块），真实目录在 epub TOC（37 条：Dedication/Part One/Chapter 1–31/...）。现有 extractor「一个 spine 文档 = 一章」→ 目录乱、章巨大还拖慢换章。

### 算法（`reader_core/extractor.py` 的 `extract_book`）
1. 读 `book.toc`（ebooklib），**扁平化**嵌套为有序列表 `[(title, href, fragment)]`（Part 这类容器条目也保留为普通条目）。
2. 若 TOC 条目数 ≥ spine 文档数 × 2（即明显比文件切分更细）才启用新路径；否则**保持现行为**（一个文档一章），保证正常打包的书零回归。
3. 新路径：逐 spine 文档 harvest 块时，同步记录每个块最近的前置锚点（沿途元素的 `id`/`name` 属性，含块自身与祖先）；按 TOC 条目的 (href, fragment) 把块序列切段；文档内 TOC 锚点之前的引导块归入上一条目（或首条目）。
4. 章标题 = TOC 条目标题；沿用现有噪音过滤（cover/copyright/目录页），另将空条目（切出 <3 块）与 "Document Outline" 这类转制垃圾条目过滤。
5. TOC 缺失/解析失败 → 完整回退现行为（try/except 包住新路径）。

### 兼容性与风险（写进 HANDOFF）
- 旧存档重开会自动重抽取（load_archive 既有行为），mockingbird 的目录即自愈；**但该书已放的书签按 {ch, ord} 锚定，切章后会漂移**——本批不做迁移，HANDOFF 里如实标注「重切章的书旧书签会漂，需手动删重放」。
- `current_chapter` 同理可能漂移，可接受。

### 验收（强标准）
- [ ] python 脚本：`extract_book('books/cb089c878a12.epub')` → 章节数 ≥ 30，标题含 'Chapter 1'…'Chapter 31'（抽查 5 个），无 'Document Outline'；每章块数分布合理（打印前 40 章标题+块数）。
- [ ] 回归：`extract_book` 跑 books/ 目录下**全部** epub，逐本打印章节数对比改前（改前先跑一遍存基线）——正常打包的书（如 The Artist's Way 27 章）章节数与标题**不变**。
- [ ] harness/preview：mockingbird 载入后目录抽屉显示正确章节序列（截图）；换章速度目测不劣化（章变小只会更快）。
- [ ] `py_compile` extractor + app；单元测试若 tests/ 有 extractor 相关用例需全过。
