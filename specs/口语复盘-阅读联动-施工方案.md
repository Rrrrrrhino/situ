# 口语复盘 · 阅读联动（复述练习）— 施工方案

> 2026-07-07 fable5 设计定盘，fable5 亲自施工（四土不派 builder）。
> 动手前必读：`~/Downloads/apply/style-codex/风格圣经.md` + 本文件。
> 与 `口语复盘-系统优化-施工方案.md`（三域一设置）兼容：复述是「输入屏的一种模式」，
> 不新增顶层域；将来 UI 大改版时本功能原样并入根屏。

## 0. 一句话

**把读过的文章变成口语题**：从最近读物里挑一篇 → 输入屏出现一张「复述题签」
（书名 + 3-5 个值得用的表达）→ 照常录音 → 复盘时 AI 知道你在复述什么、
并逐条反馈那几个表达你用上了没、用对了没。

## 1. 用户流（操作层）

1. 输入屏录音卡下方有一行克制的入口：`说一说读过的 ›`（细线框单行，像书目索引行）。
   - 手机版 / server 不可达 → 入口整行不渲染（与录音卡同一门控哲学，零回归）。
2. 点入口 → **选材屏**：最近读物列表（按日分组：日期戳 + 标题衬线 + 来源小字），
   数据来自 server RPC `/api/list_library`（桌面主窗读的同一块盘）。点一篇即返回输入屏。
3. 输入屏顶部（录音卡之下）出现**复述题签卡**：
   - 眉题小字「复 述」+ ✕（取消，回到普通复盘模式）；
   - 书名（衬线体）+ 一句静文案：「用自己的话，把它讲给一个没读过的人」；
   - **目标表达** 3-5 枚档案标签 chips（来源见 §3）。标签只是邀请，不是任务——
     文案绝不写「必须用上」。
4. 之后与现有一条龙**完全一致**：开始录音 → 停止 → chip → 点「复盘」→ 结果。
   复述态只是给复盘注入了额外上下文，不改变任何既有操作。
5. 结果页：meta 行追加 `· 复述《书名》`；现有 chunkFeedback 区自然呈现每个目标表达
   的裁决（用上/没用上/怎么改）。历史记录带上 retell 信息，回看可见。
6. 复述卡状态存 sessionStorage（与草稿同级）：中途关窗重开不丢；复盘成功后自动清除。

## 2. 数据通路（复盘窗 ↔ 阅读区）

- 复盘窗前端**不走**自己的 IndexedDB library（那是空的），直调 server RPC：
  `httpPost('/api/list_library')` / `httpPost('/api/load_archive', {id})`。
  server 的 Api 实例与主窗是**两个实例共享同一块盘**（`DATA_ROOT/library`），
  load_archive 不会碰主窗的「当前文档」状态（已核实分属两实例）。
- ⚠️ server Api 首次调用有冷启动（词表加载，1-3s）：选材屏用呼吸线加载态
  （复用读物精选骨架语法），绝不白屏。
- 手机版 Flutter 壳没有 Python server → httpPost 失败 → 入口不渲染。不碰手机代码。

## 3. 目标表达从哪来（两级来源，都不阻塞）

1. **首选：你自己点过的词**。全局生词本里 source 命中该篇的条目（星标优先，cap 5）。
   ——你读时查过的词，正是你该试着说出口的词。
2. **兜底：LLM 现场挑**。该篇生词命中 <3 个时，拿原文前 ~1500 词让 LLM 挑
   5 个「口语中高频可复用的表达」（一次小调用，prompt 见 §6）。
   **后台跑**：题签卡先上屏（书名+提示即可用），chips 位置呼吸线 → 显影；
   调用失败 = 无 chips，卡照样能练（time-to-content 铁律：辅助信息绝不挡开始）。

## 4. 复盘注入（后端零改动）

`process_dualtrack`（localapi.js）在复述态下：
- `context` = `复述练习：《书名》\n【原文开头】<前~200词>…`（拼进现有 context 参数）；
- `chunks` = 目标表达数组，**走 reviewSpeechV2 现成的 chunks 注入与 chunkFeedback 机制**
  （阶段9 已有，含「没用到的不硬凑」判定），不加新 prompt、不改 server。
- 注意：复述的目标表达**不写入词块库**（它们是临时题目）；结果页对它们的「存为词块」
  按钮照常可用（用户主动存才入库）。

## 5. 反强迫温柔（机制红线）

- **不评内容覆盖率、不打分、不列「你漏说了什么」清单**。复盘只做它一直做的事：
  语言层面的错误与更地道的说法，外加目标表达的逐条反馈。
- 目标表达没用上 → chunkFeedback 里不出现（现有机制如此），绝不批评「没完成」。
- 完成后的历史记录是描述性的：「复述《…》· 12月7日」。
- 甜点（**本批不做**，记入后续）：复述过的篇目在阅读区书架/历史上得一枚小「述」印
  （事后 bonus，圣经 §7.3 钤印模式）。

## 6. 兜底挑表达的 Prompt（新增，走主配置模型）

system：`从下面这篇英文文章里挑 5 个「中文母语学习者在口语中最值得复用」的表达
（动词短语/搭配/句式起手，不要生僻词、不要单个普通单词）。只输出 JSON：
{"chunks":[{"text":"表达原形","meaning":"简短中文义(10字内)"}]}`
user：标题 + 原文前 1500 词。maxTokens 1024，失败静默（§3 兜底逻辑）。

## 7. 施工批次与铁律

- 批A 通路：入口行 + 选材屏（server RPC + 骨架态）+ 题签卡（含 sessionStorage 持久 + ✕）。
- 批B 注入：目标表达两级来源 + process_dualtrack 复述参数 + 结果页 meta/历史标记。
- 批γ 验收：preview 闭环（选材→题签→模拟复盘注入→结果 meta；手机路径零回归）→
  亮/暗截图过圣经 §9 → `npm run build`（bundle 坑！）→ `bash packaging/build.sh` → grep baked
  → HANDOFF + CHANGELOG。
- 外科手术：主窗阅读区代码一行不碰；review.js 既有流程只加不改；
  `_effectiveDualItem`/一条龙/进度态（批⁴⑤）不许动。
