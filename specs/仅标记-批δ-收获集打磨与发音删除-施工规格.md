## 1. 各条修法（每条：现象→根因→改法→为什么→验收）

### ① 书页「收获」钮点了没反应 —— ✅已在批γ收尾窗口改好源码，本批只需装机后验证
- 现象：书页顶栏「收获」钮无反应；文章页正常。
- 根因（已坐实）：书页 `#barTools`（含 `bkSheafBtn`）每次 `renderBook` 用模板字符串重建，监听需在
  renderBook 内重挂；加载时那次 `['artSheafBtn','bkSheafBtn'].forEach` 挂在了会被替换掉的旧元素上。
  `bkCaptBtn` 因在 renderBook 内重挂（~L5330）故没坏。
- 已改：renderBook 内 `bkCaptBtn` 监听旁补 `bkSheafBtn` 监听（index.html ~L5330 已加）。
- 验收：书页点「收获」→ 右栏切到收获 tab。**本批装机后亲验这条**。

### ② 「挑几个值得练的词块」按钮偶发生成失败（点了「正在挑…」最终空，换篇重试又成功）
- 现象：复盘窗点挑词按钮，转圈后无任何词块；换一篇文章重试成功。**偶发**。
- 根因假设（低可观测，先点灯再改，memory `feedback_log_first_low_observability`）：
  `retell_targets`（mobile/js/core/localapi.js ~L1971）= ① 按 title 命中 get_global_notebook 生词
  （**书籍章节 title 常与 notebook sources.title 对不上 → 命中 0**）→ ② 不足 3 个走 `_llmPickRetellChunks`
  LLM 现挑。若 LLM 调用超时/网络抖动/JSON 解析失败 → catch 静默 → chunks 空。而
  `chunksFailed = !chunks.length && (!res || res.ok===false)`：LLM 那条 catch 掉后 res 仍可能是 `{ok:true}`
  或 retell_targets 整体 return `{ok:true, chunks:[]}` → **判成「真空则安静」→ 不显重试、静默空白**。
- 改法（分两刀）：
  1. **点灯**：`_llmPickRetellChunks` 与 `retell_targets` 的每个 catch 里 `console.warn('[retell] pick 失败:', e)`
     并在返回体带 `_pickError:true`（localapi 内部字段）。让下次失败能从 console 直接归因。
  2. **稳健兜底**：`retell_targets` 里，当 notebook 命中不足且 LLM 挑词**抛错或返回空**时，不要静默——
     retell_targets 返回 `{ok:false, error:'…'}`（而非 ok:true 空数组），使复盘窗
     `chunksFailed` 为真 → 显现有的「目标表达没取到。[重试]」（review-input.js 现成分支）。
     再加：LLM 挑词**失败重试 1 次**（同 explainer 口径）。
     **绝不**把「真的没有可挑的表达」也报成失败——只有「调用失败/异常」才 ok:false；正常挑出 0 个（罕见）仍 ok:true 静默。
- 为什么：用户点了按钮就该有确定结果——要么出词块、要么明确「没取到+重试」，绝不能转圈后无声空白（违背
  「报告结果要忠实」）。
- 验收：断网点挑词 → 显「没取到+重试」而非空白；正常点 → 出 3–5 枚；console 有失败归因日志。

### ③ 词块发音小喇叭（用户新诉求·痛点）
> 用户看到词块时「很想听发音」。基建现成：主窗 `speak(word)`+`api().get_audio({word,accent})`；
> 复盘窗 `playAudio(word, btn)`+`api('get_audio')`。get_audio 后端 MiniMax 整句/词块都能播（app.py L1646）。
- **两处都加**：
  a. **复盘窗 复述题签卡 chips**（review-input.js `_loadRetellSlot` 的 chips 渲染）：每枚 chip 内或旁加一个
     小喇叭（复用 `.gv-say` 同款细线 SVG，17×17，muted→accent hover），点击 `playAudio(c.text, 喇叭元素)`，
     **`stopPropagation` 不触发 chip 展开**（参照 app.js L2536「🔊 play only never expand」）。
  b. **阅读页右栏「收获」区 每条目**（index.html `_shEntryHtml`）：词块标题 `.sh-chunk` 旁加小喇叭，
     点击 `speak(it.chunk)`；`stopPropagation` 不触发别的。导出的自包含 HTML **不带**喇叭（无 JS，静态）。
- 发什么音：**整个词块**（chunk）发音（MiniMax 对短语/句子友好；单词也能播）。不做「重点词汇 vs 整块」二选一
  开关——一个喇叭播整块即可（少写阶梯：一个按钮解决，别加选择器）。
- 图标：细线小喇叭，复用现有 `GV_SVG.say`（index.html ~L2513）那条 path，stroke-width 1.7 同族，
  **绝不用 🔊 emoji**（memory `feedback_print_aesthetic_no_gloss`）。
- 验收：点复述 chip 喇叭 → 出声、chip 不展开；点收获条目喇叭 → 出声；无 key 时 get_audio 失败要静默不报错爆屏。

### ④ 换章后点「收获」里前章条目的原文小注 → 回不去（左侧提示「先翻页」）
- 现象：收获区某条属前面章节，点它的「原文·第N章」小注，界面提示先翻页、不自动定位。
- 根因：`_shScrollToSrc`（index.html）当 `ch!==bookIdx` 时直接 return + setStatus 提示。
- 改法：跨章时**自动跳章再滚**——调现有跳章函数（找 `loadChapter`/`gotoChapter`/目录点击项走的那条，
  书模式换章入口，grep `_load_chapter`/`renderBook` 的章节切换），跳到 `it.ch` 章，等该章渲完
  （章节渲染是异步/需等 `_applyCapturesForChapter` 那轮）后再 `_bkBlocks()[blockOrd]` 滚动+呼吸高亮。
  实现口径参照书签跳转（书签点击能跨章定位，找 `_bkGoToBookmark`/书签 list 点击处照抄它的「跳章+滚到锚点」）。
- 为什么：收获集的核心价值就是「对照原文」，跨章回不去=功能残缺；且用户明确「不必要，应自动定位」。
- 验收：在第 3 章，点收获里第 1 章某条的原文小注 → 自动翻到第 1 章并滚到那句、呼吸高亮一次。

### ⑤ 「收获」区排序不严格按章节 + 点新词块后收获区不自动刷新
- 现象 a：按章节序时，顺序不严格跟章节走。
- 现象 b：仅标记下点新词块，收获区没自动更新（要重开 tab 才见）。
- 根因：a）`_shOrder` 排序键 `(ch, blockOrd, ts)`——核实 book 模式 `blockOrd` 是否章内块序（应是），
  若仍乱，加三级键 `start`（capture 锚点偏移）：`(ch, blockOrd, start, ts)`；并确认 ch=null（文章）与
  数字（书）不混排。b）收获 tab 只在 `openSheaf()`（tab 打开时）与「有 pending 时的 3s 轮询」里刷新——
  纯新增/取消不触发重渲。
- 改法：
  a. 排序键补 `start`（收获 item 需带 `start`——批α 落盘时把 capture 的 start 一并写进 item；或前端从
     `_captures` 反查补上）。**注意**：这要动批α 落盘 schema 加 `start` 字段 → 属跨批改动，若不想动 α，
     退而用前端 `_captures` 反查 start 排序（item.cid→_captures.find.start）。**优先前端反查，不动 α 落盘**。
  b. 收获 tab 打开状态下，`captureRange`/`uncapture` 成功后：若 `panelSh` 可见，防抖 400ms 调一次
     `openSheaf()`（或轻量 `_shRender(await get_sheaf)`），新条目显骨架、删掉的消失。防抖避免连划多条狂刷。
- 为什么：用户「边划边攒、读完即得」，正看着收获区却不随手更新=违和；排序错乱破坏「对照原文」的定位直觉。
- 验收：收获 tab 开着划一条 → 数秒内该条以骨架出现、生成后 ink-in；顺序严格按 章→块→偏移。

### ⑥ 取消划痕后，收获区没同步删掉那条（幽灵条目）
- 现象：仅标记下点掉一条划痕，收获区那条还在。
- 根因：删除同步只在 `_run_sheaf`（生成触发时）跑；纯取消不达生成阈值 → 不触发 → sheaf 落盘仍留旧 item；
  `get_sheaf` 原样返回。
- 改法（**惰性删除同步，最省**）：`get_sheaf` 返回前，用当前 library `captures` 的活 cid 集合过滤
  `items`/`pending`（app.py `get_sheaf` 里加一步：读 `_load_captures(doc_id)`→`_merge_captures`→live cid 集，
  `items=[it for it in items if it.cid in live]`，pending 同理）。这样**即使没触发生成，也永不返回幽灵条目**，
  且不落盘（只影响返回视图；真正落盘删除仍由下次 `_run_sheaf` 收口）。配合 ⑤b 的前端刷新，取消即从界面消失。
- 为什么：删了还在=数据不一致，用户会不信任这个面板。惰性过滤是「少写阶梯」最优解（不新增写盘、不改触发逻辑）。
- 验收：仅标记下取消一条 → 收获区（开着的话）那条随刷新消失；重开 tab 也不再有它。

### ⑦ 讲解模式下：点过/收藏过的单词与词块，没有删除入口（用户要）
> 后端已有 `delete_global({key})`（app.py L1466，从全局生词本移除该 key）。前端只在**生词本大视图
> `gvWrap` 且 `_gv.scope==='global'`** 才渲染 `.gv-del`（index.html L2642/L2719）。缺口=右栏「生词本」tab
> （`panelNb`）与「本篇」范围没有删除。
- 改法：
  a. **右栏生词本 tab（panelNb 的 `showNotebook` 渲染）**每行加一个 `.gv-del` 小垃圾桶（细线 SVG，复用
     `GV_SVG.del`），点击 → `api().delete_global({key})` + 从内存 notebook 移除该条 + 重渲 + 刷新 `nbCount`。
     「本篇」会话生词本（`self._notebook`）也要能删：若 delete_global 只删全局，加一个后端
     `delete_notebook_entry({key})` 从 `self._notebook` 移除（或 delete_global 顺带清 session 镜像）——
     先查 delete_global 是否已连带清 session，没有就补一个 session 删除的桥。
  b. 生词本大视图「本篇」范围也放开 `.gv-del`（去掉 `_gv.scope==='global'` 的门控，或本篇走 session 删除桥）。
- 交互：删除用**应用内确认框**（`cfmWrap` 现成，index.html L1285「删除确认」那套），别用原生 confirm；
  单条删除也可直接删+可撤销 toast（二选一，倾向直接删+status 提示「已移除 X」，与生词本大视图现有 gv-del 一致）。
- 为什么：能收藏不能删=只进不出，词表迟早污染；用户点名要。
- 验收：右栏生词本 tab 某行点垃圾桶 → 该词消失、nbCount-1、重开仍不在；本篇与全局都能删。
