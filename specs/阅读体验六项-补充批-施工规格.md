## A. 讲解区单词按 A–E 频率档着色（+ 短语/句子里的重点词汇）

**现状**（`index.html`）：
- 词讲解 `explanationHtml`（L1536）：标题 `.ex-word` 里的词是 `color:var(--fg)`（L158 深色），旁边已有 `.ex-freq` 档位角标（L1547，数据 `r.freq_band` 现成）。
- 短语/句子 `selectionHtml`（L1557）：重点词汇 `.ex-kw li` 每条 `<b>${k.word}</b> <span class="g">${k.gloss}</span>`（L1560-1562），`<b>` 是赭色 `--accent`（L195）。**每个 key word 不带频率数据**。

**改法**：
1. **词标题上色**（数据现成，简单）：`explanationHtml` 里 `.ex-word` 的词 span 按 `r.freq_band` 染频率色。做法：给该 span 内联 `style="color:var(--fX)"`（X=A..E，映射 `{A:'--fA'...E:'--fE'}`，与生词本/全局同一套），无 band 时不染（回落 `--fg`）。角标保留。
2. **重点词汇上色**（需后端补分类）：
   - **后端 `app.py explain_selection`**（L670）：拿到 `res` 的 `key_words` 后，用现成 classifier 给**单词型** key 补频率档。classifier 取法同 `explain_word`：`classifier = self._last["classifier"] if self._last else None`（书模式/文章模式 `self._last` 都有 classifier，已核实）。对每个 `k`（`res["key_words"]`），若 `k["word"]` 是单个词（`len(k["word"].split())==1` 且全字母）：`info=classifier.classify_word(k["word"], sentence)`，`rank=info.get("daily_rank")`，`band=_freq_band(rank)`，命中则 `k["freq_band"]=band["label"]`。多词固定搭配不给 band（保持现有赭色 `<b>` 强调即可——它们本就是"标出来"的搭配）。**这段包在 try/except，classifier 缺失/异常就跳过，不影响讲解主体**。缓存写入的 base 也带上补好 band 的 key_words（回看一致）。
   - **前端 `selectionHtml`**（L1560）：渲染每个 key word 时，若 `k.freq_band` 存在，给 `<b>` 内联 `style="color:var(--fX)"` 覆盖默认赭色；无 band 的多词搭配保持赭色 `<b>` 不变。
3. **暗色模式**：`--fA..E` 已有暗色变体（L970 `.dark` 段），内联 `var(--fX)` 自动跟随，无需额外处理。

**验收**：preview 注入一个词讲解（带 freq_band=A/C/E）→ `getComputedStyle('.ex-word span')` 颜色＝对应 `--fX`；注入一个句子讲解（key_words 里混单词+多词搭配，单词带 freq_band）→ 单词 `<b>` 染频率色、多词搭配仍赭色。Python 侧：`explain_selection` 对含单词 key 的选区返回的 `key_words` 带 `freq_band`（构造 mock classifier 或真跑一段）。截图过眼：讲解区配色与生词本一致、暗色和谐。

## B. 读物精选：点入口后改「浮层卡片」，不再向下顶出屏幕

**现状**：`#discList`（L1113，`<div class="disc-list" style="display:none">`）是**块级、就地展开在来源卡下方**（`renderDiscList` L3866 填充、`toggleDisc` L3880 控制显隐），文章多时把首页顶出当前屏。

**改法**：把 `#discList` 改成**锚定在被点来源卡附近的浮层卡片（popover）**：
- 定位：相对被点的 `.disc-src` 浮出——**优先浮在该卡上方**（`bottom` 对齐卡片顶、留 8px 间隙）；若上方空间不足（卡片离视口顶太近）则浮到**下方**；水平左对齐该卡左缘，超右边界则右对齐。用 `position:absolute`（挂在 `.disc-band` 或 `.home-wrap` 这个 `position:relative` 容器上，用被点卡片的 `offsetLeft/offsetTop/offsetHeight` 算坐标），或 `position:fixed` + `getBoundingClientRect()`（二选一，`fixed`+rect 更简单不受滚动影响，但要在滚动/resize 时关闭 popover 避免错位——推荐点开即定、滚动就关）。
- 卡片样式：白纸卡 `--paper` + 细边 `--border` + 柔和投影（复用 clip/gv 卡的阴影语言）；**固定 `max-height`（如 min(60vh, 420px)）+ 内部 `overflow-y:auto`**，这样再多文章也只在卡内滚、绝不顶出屏幕；`min-width` ~320、`max-width` ~420；圆角 10。
- 顶部保留来源名 + feed 说明 + `↻刷新` + `✕关闭`（`收起`可换成 ✕）；点卡外空白/Esc 关闭；再次点同来源卡也关闭（toggle）。
- 打开时给被点 `.disc-src` 加 `.on` 高亮（已有样式 L935）。
- z-index 要压过书架/剪报但别挡住全局弹层（参考现有 overlay 的层级，取一个介于内容与模态之间的值）。

**验收**：preview 首页点某来源→popover 浮现（`getBoundingClientRect` 量出它 relative 到来源卡的位置：上方优先、卡内可滚、`max-height` 生效）；书架/剪报**零位移**（量宽高/位置前后不变）；文章列表很长时页面 body 不出现纵向溢出把首页顶走；点 ✕/卡外/Esc 关闭；暗色截图和谐。给用户的视觉证明复制一份到 `~/Desktop/预览/四土/`。

## C. 追问：切回某词时，恢复的追问答案默认「展开」（不再折叠）

**现状**：`mountExplanation`（L1666）恢复 `r.followups` 历史时每条渲成 `<div class="ask-q collapsed">…</div><div class="ask-a">…</div>`（L1679，**故意折叠**）。用户从 a→b→回 a 就是想直接看 a 的答案，折叠等于多一步点击。

**改法**：把 L1679 恢复历史时的 `ask-q collapsed` 改为 `ask-q`（**默认展开**，答案 `.ask-a` 直接可见）。点问句气泡仍可手动折叠（`.ask-q` 点击 toggle 逻辑 L1759 不动）。
- 注意：这只影响「切回/重开时恢复的历史」。sendAsk 新发的追问本就是展开的，不受影响。
- 若担心历史很多时全展开太长：**只需全部展开即可**（用户明确要一眼看到答案，不要折叠）；不做"只展开最后一条"的取巧。

**验收**：preview 复现 a→追问→切 b→切回 a：`.ask-a` 可见（不含 `.ask-q.collapsed`）、答案文字直接显示，无需再点。手动点问句仍能折叠。

## D. 读物精选提速：The Conversation 只拉 global + 结果缓存 + 首页预取

**现状**：`renderDiscList`（L3866）`Promise.all(outlet.feeds.map(...))` 对 The Conversation 并行拉 **3 个 feed（global/us/uk）** 再合并，等最慢的那个；无任何缓存，每次点都重新网络请求。文章点开走 `process(url)` 抓正文+spaCy 分词，是网络+NLP 固有耗时（本批不强改，见下）。

**改法**：
1. **The Conversation 砍成单 feed**：`DISC_OUTLETS`（L3837）里 The Conversation 的 `feeds` 只保留 `{name:'全球', url:'.../global/articles.atom'}`（删 us/uk）。一次请求，快 & 稳。
2. **结果内存缓存**：加 `const _discCache=new Map();`（outletId → items）。`renderDiscList` 先查缓存命中则**立即渲染**（0 延迟），后台可选静默刷新；未命中才 `fetch_feed` 并写缓存。`↻刷新` 按钮强制绕过缓存重拉。
3. **首页预取**：`renderDiscOutlets`（首页渲染时，L3846）之后，用一个 `requestIdleCallback`/`setTimeout` **后台预取所有来源**（每个 `fetch_feed` 写入 `_discCache`，`.catch` 静默失败），这样用户点任意来源时多半已是缓存命中、秒开。预取只在首页且每次会话首次进首页时跑一遍（避免重复轰炸）。
4. **文章解析耗时**：属网络抓取 + spaCy 分词固有成本，本批**不做激进优化**（避免动核心分词管线的风险）。但要确保点文章后**立即有加载反馈**（现有 `process` 的 `setStatus` 阶段提示"① 抽取正文… ② 词汇分层…"应可见；若 popover 场景下被挡，补一个"正在解析…"态）。若后续仍嫌慢，另开一轮专门优化（可讨论 hover 预取/spaCy 模型裁剪），本批先把 feed 侧的等待消掉。

**验收**：preview 点 The Conversation → `preview_network` 只发 **1 个** feed 请求（非 3 个）；第二次点同来源**无新网络请求**（缓存命中）、列表瞬间出；`↻刷新`触发一次新请求。首页加载后隔一会儿 `_discCache` 已含各来源（预取生效）。CNA 行为不变。

## E. MiniMax 发音 key 分发 —— 已核实无需改（仅记录）

核实结论（**本批不动代码**，写在这里备查、并请 builder 在汇报里复述以免遗忘）：
- 设置弹层已有 `#setMmKey`/`#setMmGroup` 输入（index.html L1222-1223），标注"选填·留空用有道"。
- 基础发音走**有道**（`_download_audio`，无需任何 key），装了即可发音；MiniMax 是可选升级，朋友自填自己的 key/group。
- key 本就不应打进分发包（安全铁律），故分发包不带 key 是正确的。
- **结论：发音不会因分发而失效，此项无需改动。** 若日后想让分发包默认就有更好发音，只能是"引导朋友填自己的 MiniMax key"或"提供一个共享 key"（涉及成本/额度，需用户决定），不在本批。
